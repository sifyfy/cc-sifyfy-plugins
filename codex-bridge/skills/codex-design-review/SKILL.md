# Codex設計レビュースキル

## 概要

設計フェーズでCodexをセカンドオピニオンとして活用する。
Claudeが提案した設計に対し、Codexに独立した代替案や批評を求め、
両者の設計案を比較してシフィさんに提示する。

## コンテキスト共有の原則

「事実は共有、解釈は独立」を遵守する:
- **共有してよい**: 要件、制約条件、既存コードの構造、技術スタック、非機能要件、期限
- **共有しない**: Claudeの結論、分析結果、推奨案（Devil's Advocate除く）

## 実行手順

### 1. Claudeによる設計案の整理

まずClaude自身の設計案を以下の要素で整理する（内部保持）:

- アーキテクチャ概要
- 主要なコンポーネントと責務
- データフロー
- 採用した設計パターンとその理由
- トレードオフ

### 2. Codexセッションの開始

`mcp__codex__codex` を呼び出す。developer-instructionsとプロンプトは `references/prompt-templates.md` を参照。

- **cwd**: 現在のプロジェクトディレクトリ
- **sandbox**: `read-only`

重要: Claudeの設計案はCodexに渡さない（独立性を確保）。

### 3. 深掘り

必要に応じて `mcp__codex__codex-reply` で:
- 特定のコンポーネントについて詳細を質問
- トレードオフの分析を依頼
- スケーラビリティやパフォーマンスの観点を深掘り

### 4. 設計案の比較と提示

```
═══ 設計レビュー結果（Claude + Codex セカンドオピニオン）═══

【要件】
  {requirements_summary}

【Claudeの設計案】
  アーキテクチャ: ...
  主要コンポーネント: ...
  設計パターン: ...
  メリット: ...
  デメリット: ...

【Codexの設計案】
  アーキテクチャ: ...
  主要コンポーネント: ...
  設計パターン: ...
  メリット: ...
  デメリット: ...

【設計上の一致点】
  - 両者が同意する設計判断

【設計上の相違点】
  | 観点 | Claude | Codex |
  |------|--------|-------|
  | ... | ... | ... |

【推奨】
  → ハイブリッド案（両者の良い点を組み合わせ可能な場合）
  → シフィさんの判断を仰ぐ点（相違がある場合）
```

### 5. ハイブリッド設計案の検討

両者の設計から良い要素を抽出してハイブリッド案を提案できる場合は提案する。

以下の条件に該当する相違のみシフィさんにエスカレーションする:
- 仕様解釈が変わる
- 保守コストに大差がある
- 性能・安全性に影響する
- 破壊的変更を含む

上記に該当しない軽微な差分はClaudeが統合判断してよい。

## 注意事項

- Claudeの設計案をCodexに見せない（独立性を確保）
- Codexの設計案が既存コードベースと乖離する場合はその旨を明記
- SOLID原則やDRY/KISSの観点も比較に含める
- **環境差異への注意**: Codexのサンドボックス環境と実環境（OS、ランタイム、CI/CD）が異なる場合、Codexの設計案が実環境で機能しない可能性がある。環境固有の制約はClaudeが補完する
- Codexは「暗黙の前提を洗い出す」のに特に有効。設計の前提条件を明示的に質問すると質の高い分析が得られる
