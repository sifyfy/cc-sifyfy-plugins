# Codex要件分析スキル

## 概要

要件分析フェーズでCodexをセカンドオピニオンとして活用する。
Claudeが自身で要件を分析した上で、Codexに独立した視点で分析を依頼し、
両者の見解を統合してシフィさんに提示する。

## コンテキスト共有の原則

「事実は共有、解釈は独立」を遵守する:
- **共有してよい**: 要件、制約条件、既存コードの構造、技術スタック、非機能要件、期限
- **共有しない**: Claudeの結論、分析結果、推奨案（Devil's Advocate除く）

## 実行手順

### 1. Claudeによる要件分析

まずClaude自身が以下を分析し、内部的にまとめる:

- 要件の明確さ・完全性
- 暗黙の前提条件
- 潜在的なリスクや矛盾
- スコープの曖昧な点
- 不足している非機能要件

### 2. Codexセッションの開始

`mcp__codex__codex` を呼び出す。developer-instructionsとプロンプトは `references/prompt-templates.md` を参照。

- **cwd**: 現在のプロジェクトディレクトリ
- **sandbox**: `read-only`

重要: Claudeの分析結果はCodexに渡さない（独立性を確保）。

### 3. 深掘り

必要に応じて `mcp__codex__codex-reply` で特定の要件について詳細を質問する。

### 4. 見解の統合と提示

```
═══ 要件分析結果（Claude + Codex セカンドオピニオン）═══

【分析対象】
  {requirements_summary}

【Claudeの分析】
  - ポイント1
  - ポイント2

【Codexの分析】
  - ポイント1
  - ポイント2

【一致した指摘】
  - 両者が共通して指摘した点

【相違点・追加の観点】
  - Claudeのみ: ...
  - Codexのみ: ...

【要確認事項】（相違がある場合）
  → シフィさんの判断を仰ぐ項目
```

### 5. エスカレーション

以下の条件に該当する相違のみシフィさんにエスカレーションする:
- 仕様解釈が変わる
- 保守コストに大差がある
- 性能・安全性に影響する
- 破壊的変更を含む

上記に該当しない軽微な差分はClaudeが統合判断してよい。

## 注意事項

- Codexはread-onlyサンドボックスで動作させる
- 両者の見解を公平に提示し、Claudeの見解を優先しない
- Codexの分析が抽象的な場合は `codex-reply` で具体的なシナリオや根拠を求める
- 実行環境固有の要件（OS依存、ハードウェア制約等）についてはCodexの分析精度が下がる場合がある。Claudeが実環境の知識で補完する
