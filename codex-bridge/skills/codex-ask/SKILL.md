# Codex汎用質問スキル

## トリアージ

まずユーザーの質問が特化スキルに該当するか判断する:

- 要件分析 → `codex-bridge:codex-requirements-analysis` スキルを使用
- 設計・アーキテクチャ → `codex-bridge:codex-design-review` スキルを使用
- コードレビュー → `codex-bridge:codex-code-review` スキルを使用
- アルゴリズム → `codex-bridge:codex-algorithm-design` スキルを使用
- Claudeの提案への反論が欲しい → `codex-bridge:codex-devils-advocate` スキルを使用
- 上記に該当しない → このスキルで対応

## コンテキスト共有の原則

「事実は共有、解釈は独立」を遵守する:
- **共有してよい**: 要件、制約条件、既存コードの構造、技術スタック、非機能要件、期限
- **共有しない**: Claudeの結論、分析結果、推奨案（Devil's Advocate除く）

## 実行手順

### 1. Claudeによる分析

ユーザーの質問について、まずClaude自身が分析・回答を内部的にまとめる（まだユーザーに見せない）。

### 2. Codexセッションの開始

`mcp__codex__codex` を以下のパラメータで呼び出す:

- **developer-instructions**:

```
あなたはセカンドオピニオンを提供する独立したアドバイザーです。
別のAIアシスタント（Claude）も同じ質問に取り組んでいます。
あなたはClaudeの回答を知りません。独立した視点で回答してください。

- 独立した視点で分析する
- 前提を疑い、批判的に分析する
- 同意するだけでなく、異なるアプローチを積極的に検討する
- 根拠を明確にして回答する
- 以下のフォーマットで回答する: 前提 / 提案 / リスク / 捨てた案 / 確信度
- 日本語で回答する
```

- **prompt**: ユーザーの質問を整形し、事実コンテキスト（要件、制約、関連コード、プロジェクト情報）を付加。Claudeの結論は含めない
- **cwd**: 現在のプロジェクトディレクトリ
- **sandbox**: `read-only`

### 3. 深掘り（必要に応じて）

Codexの応答が不十分な場合は `mcp__codex__codex-reply` で補足質問する。
`threadId` はセッション開始時の戻り値から取得する。

### 4. 見解の統合と提示

以下のフォーマットでシフィさんに提示する:

```
═══ Codex セカンドオピニオン ═══

【質問】
  {質問の要約}

【Claudeの見解】
  - ポイント1
  - ポイント2

【Codexの見解】
  - ポイント1
  - ポイント2

【一致点】
  - 両者が共通して挙げた点

【相違点】
  - Claudeのみ: ...
  - Codexのみ: ...

【要確認事項】（相違がある場合）
  → シフィさんの判断を仰ぐ項目
```

### 5. エスカレーション判断

以下の条件に該当する相違のみシフィさんにエスカレーションする:
- 仕様解釈が変わる
- 保守コストに大差がある
- 性能・安全性に影響する
- 破壊的変更を含む

上記に該当しない軽微な差分はClaudeが統合判断してよい。

## 注意事項

- Codexの応答を無批判に受け入れない。根拠が不明な場合は `codex-reply` で確認する
- 両者の見解を公平に提示し、Claudeの見解を優先しない
- Codexが日本語以外で応答した場合は、Claudeが翻訳して統一フォーマットで提示する
- Codexはread-onlyサンドボックスで動作するため、コードの実行検証はできない。指摘の妥当性はClaudeが検証する
- Codexの応答が表面的・抽象的な場合は `codex-reply` で具体例や根拠を求めることで質を高められる
