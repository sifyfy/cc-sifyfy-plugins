# Codexコードレビュースキル

## 概要

コードレビューでCodexをセカンドオピニオンとして活用する。
Claudeが自身でレビューした上で、Codexに独立したレビューを依頼し、
両者の指摘を統合してシフィさんに提示する。

## コンテキスト共有の原則

「事実は共有、解釈は独立」を遵守する:
- **共有してよい**: 要件、制約条件、既存コードの構造、技術スタック、非機能要件、期限
- **共有しない**: Claudeの結論、分析結果、推奨案（Devil's Advocate除く）

## 実行手順

### 1. Claudeによるコードレビュー

対象コードを以下の観点でレビューし、結果を内部的にまとめる（まだユーザーに見せない）:

- コード品質（可読性、保守性）
- バグの可能性
- セキュリティ上の懸念
- パフォーマンスの問題
- SOLID原則、DRY/KISS、immutability優先への準拠

### 2. Codexセッションの開始

`mcp__codex__codex` を呼び出す。developer-instructionsとプロンプトは `references/prompt-templates.md` を参照。

- **cwd**: 現在のプロジェクトディレクトリ
- **sandbox**: `read-only`

重要: Claudeのレビュー結果はCodexに渡さない（独立性を確保）。

### 3. 深掘り

Codexの指摘が抽象的な場合は `mcp__codex__codex-reply` で具体的なコード行や修正案を求める。

### 4. レビュー結果の統合と提示

```
═══ コードレビュー結果（Claude + Codex セカンドオピニオン）═══

【レビュー対象】
  ファイル: {files}

【両者が一致して指摘した問題】（重要度順）
  1. [重大] {issue} (ファイル:行)
     → 修正案: ...

【Claudeのみが指摘した問題】
  1. [{severity}] {issue}

【Codexのみが指摘した問題】
  1. [{severity}] {issue}

【見解が異なる点】
  - {point}: Claude={opinion} / Codex={opinion}
    → シフィさんの判断を仰ぎます

【総合評価】
  Claude: {overall}
  Codex: {overall}
```

### 5. エスカレーション

- 「両者が一致して指摘」は信頼度が高い指摘として強調する
- セキュリティ関連の指摘はどちらが発見したかに関わらず最優先

以下の条件に該当する相違のみシフィさんにエスカレーションする:
- 仕様解釈が変わる
- 保守コストに大差がある
- 性能・安全性に影響する
- 破壊的変更を含む

上記に該当しない軽微な差分はClaudeが統合判断してよい。

## Codexの得意・苦手

| 観点 | 評価 | 補足 |
|---|---|---|
| エッジケースの発見 | 得意 | Claudeが見落としがちなパターンを指摘できる |
| 機械的な品質チェック（命名、重複等） | 得意 | |
| プロジェクト固有の慣例・文脈 | 苦手 | コードベース全体の把握が限定的 |
| 環境固有の問題（OS差異等） | 苦手 | サンドボックス環境と実環境の差異がある |
| セキュリティの基本パターン | 普通 | OWASP系の定型チェックは可能 |

## 注意事項

- 両者の指摘が矛盾する場合は、根拠を添えて両方を提示
- Codexが不正確な行番号を指摘する場合があるため、Claudeが検証して補正する
- Codexの指摘が抽象的（「改善の余地がある」等）な場合は `codex-reply` で具体的なコード行や修正案を求める
- プロジェクト固有の慣例に反する指摘は、慣例の存在をClaudeが補足してシフィさんに提示する
