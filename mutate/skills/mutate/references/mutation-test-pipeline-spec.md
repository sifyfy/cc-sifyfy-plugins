# Mutation Test Pipeline

## 目的（Why）

AIエージェントが生成したテストの品質を、ミューテーションテストという客観指標で検証し、反復的に改善するパイプラインを提供する。

従来のAIテスト生成では、カバレッジが高くても実際のバグ検出能力が低い「トートロジカルテスト」が量産される問題がある。カバレッジやAIの自己申告に依存せず、「コードを意図的に壊したときテストが本当に落ちるか」という客観的事実に基づいてテスト品質を評価・改善する。

## 振る舞い（What）

### パイプライン全体の流れ

パイプラインは3つのフェーズから構成される。

#### フェーズ1: ベースライン確立

| Given                      | When                                 | Then                                                    |
| -------------------------- | ------------------------------------ | ------------------------------------------------------- |
| specと実装コードが存在する | パイプラインを起動する               | テスト生成エージェントがspec+コードからテストを生成する |
| テストが生成された         | テストを実行する                     | 全テストがパスすることを確認する                        |
| テストがパスした           | ミューテーションテストを初回実行する | ベースラインのmutation scoreを記録する                  |
| テストがパスしない         | —                                    | エラーを報告し、テスト修正を試みる（最大3回）           |

#### フェーズ2: Mutation-Driven Test Refinement

| Given                                          | When                                         | Then                                                        |
| ---------------------------------------------- | -------------------------------------------- | ----------------------------------------------------------- |
| ベースラインのmutation scoreが閾値未満         | refinementループに入る                       | surviving mutantsを分類・要約してレビューエージェントに渡す |
| レビューエージェントがフィードバックを生成した | フィードバックをテスト生成エージェントに渡す | 追加テストを生成する（実装コードの変更は禁止）              |
| 追加テストが生成された                         | ミューテーションテストを再実行する           | mutation scoreの改善量を記録する                            |
| mutation scoreが閾値以上になった               | —                                            | パイプラインを成功終了する                                  |
| 改善量が閾値未満の状態がN回連続した            | —                                            | フェーズ3へ遷移する                                         |

#### フェーズ3: Design Escalation

| Given                           | When                 | Then                                                      |
| ------------------------------- | -------------------- | --------------------------------------------------------- |
| フェーズ2で改善が頭打ちになった | 設計診断モードに入る | surviving mutantsのパターンから設計上の阻害要因を分析する |
| 設計問題が特定された            | —                    | リファクタリング提案を生成して人間に提示する              |
| 設計問題が特定されなかった      | —                    | 残存mutantsの分類結果とともにパイプラインを終了する       |

### ループ停止条件

| 条件                                              | 結果                          |
| ------------------------------------------------- | ----------------------------- |
| mutation score ≥ 閾値（デフォルト: 80%）          | 成功終了                      |
| 直近3回のループでmutation scoreの改善が各回2%未満 | フェーズ3へ遷移（または終了） |
| surviving mutantsの過半数が等価mutant候補         | 成功終了（実質的な上限到達）  |
| surviving mutantsの過半数が設計起因               | フェーズ3へ遷移               |
| ループ回数が上限（デフォルト: 5回）に到達         | 結果レポートとともに終了      |

### 制約・不変条件

- **テスト生成フェーズでの実装変更禁止**: mutation scoreの改善がテスト品質向上によるものか実装変更によるものか区別できなくなるため
- **mutation reportの情報圧縮**: 1回のフィードバックにつき最大10件の高優先度mutantsに絞る
- **等価mutantの過剰追及禁止**: コスト効率が急激に悪化するため
- **mutation scoreの絶対視禁止**: 仕様レビューの代替ではない
- **既存テストの削除禁止**: 既存テストにはパイプラインが把握できない意図がある可能性がある
